The core insight

The score is bits per byte, not per token. That makes the decomposition explicit:

bpb = (cross-entropy in bits per token) / (bytes per token)

The starter's byte-level tokenizer has bytes_per_token = 1.0, so it forfeits the entire denominator. Training a byte-level BPE on the provided corpus raises that to 3.152 bytes/token on dev. The per-token cross-entropy necessarily rises — the model now picks among 3,072 options instead of 256 — but it would have to get more than 3.15× worse to break even, and it does not. Most of the improvement over the baseline comes from this one change; everything else is spending the remaining budget well.

There is a second, compounding effect. Devanagari is 3 bytes per character in UTF-8, so the baseline burns three tokens per Hindi character and its 128-token window covers roughly 43 Hindi characters. After BPE, common Hindi syllables collapse into single tokens and a 512-token window covers about 1,600 bytes of genuine context.

Depth was preferred over width: at fixed parameters, 6×144 outperformed the wider, shallower shapes that fit the same budget. Vocab 3072 is the deliberate knee of the compression curve — 6144 and 8192 compress better (3.44 and 3.56 bytes/token) but the embedding matrix grows linearly and starves the layers that do the actual computation.

Ordering matters and is not arbitrary: R3's 3e-3 learning rate is only stable because R2 fixed the initialisation and added clipping. Run in the other order, the high LR diverges — which is exactly the kind of interaction a one-factor-at-a-time sweep hides.

Verification

A lossy tokenizer makes bpb meaningless and disqualifies the run, so decode(encode(x)) == x is checked on the full dev file plus emoji, CJK, raw control bytes, and the empty string. sanity.py also confirms the parameter cap, that odd-length and full-block_size windows both forward correctly, and that the official command runs unmodified:

python sanity.py
python evaluate.py --checkpoint ckpt.pt --text_file <any_text_file>

RoPE was verified numerically rather than assumed: with the implemented interleaved rotation, dot(rope(q,m), rope(k,n)) is identical for every pair at the same offset m-n, confirming it encodes purely relative position.

Two specific cases worth naming. First, the assistant's initial BPE encoder rescanned every pair on every merge — O(n²) per piece — and hung on the full dev file; it was caught by timing the encode step and replaced with a linked-list plus heap implementation that encodes the dev file in 0.06 s. Encoding speed is not cosmetic here: the scorer encodes the entire hidden file. Second, the first parameter-budget calculation was simply wrong and reported feasible configurations that were 2–3× over the cap; the corrected accounting is the table above, and it is what drove the choice of vocab 3072 over 6144.

Honest limitations

  • The dev/hidden gap is unmeasured — BPE merges are fit to train_corpus.txt, so compression on the hidden file will be slightly worse than 3.152 bytes/token if its English/Hindi mix differs.
  • The vocab knee at 3072 was chosen from the compression curve and the parameter cap, not from a full bpb sweep at every vocab size; a longer budget would test 2048 and 4096 end-to-end.
  • Single seed. Runs at this scale vary run-to-run, and no variance estimate was collected.
